# Day / Night 主题切换设计

## 背景

当前启动器界面已经稳定为一套夜间风格（night version），但产品接下来进入快速迭代阶段，需要同时维护一套白天风格（daily / day version），并允许用户直接在界面里切换主题。

当前代码状态具备做这件事的基础：

- 壳层集中在 `AppShell`
- 左侧导航里已有“灯泡”入口
- 样式已经拆分为 `tokens.css`、`shell.css`、`home.css`、`settings.css`

## 目标

本次调整目标只有三个：

1. 提供 `night` / `day` 两套主题
2. 点击侧边栏“灯泡”即可切换主题
3. 主题选择写入 `localStorage`，下次打开时自动恢复

## 已确认决策

### 主题集合

- 只做两套主题：
  - `night`
  - `day`

### 默认行为

- 启动时优先从 `localStorage` 读取主题
- 读取不到时默认使用 `night`

### 切换入口

- 左侧导航里的“灯泡”不再作为普通页面入口
- 点击后直接切换主题
- 这次不再让它切到一个“灯泡页面”

### 持久化策略

- 只使用前端 `localStorage`
- 这次不接 Tauri 配置文件或 Rust 侧持久化

## 架构方案

### 1. 状态位置

主题状态放在 `AppShell`。

原因：

- 主题影响整个应用壳层与所有页面
- `AppShell` 已经是当前页面切换和整体布局的最外层
- 不需要为这次主题切换额外引入全局状态库

建议状态结构：

- `theme: 'night' | 'day'`

### 2. DOM 挂载方式

在最外层根节点上挂 `data-theme`。

例如：

- `data-theme="night"`
- `data-theme="day"`

原因：

- 最适合让 CSS variables 统一接管
- 不需要在每个组件里单独传主题 prop
- 后续如果扩展第三套主题，也只需要继续加 token

### 3. 样式来源

主题差异优先通过 `tokens.css` 的变量控制。

做法：

- 保留当前夜间主题作为 `night`
- 在 `tokens.css` 中补一套 `day` 变量
- `shell.css`、`home.css`、`settings.css` 中优先把关键硬编码颜色改成变量引用

## 本次要覆盖的视觉范围

第一版主题切换至少要覆盖这些区域：

1. 整体背景
2. 侧边栏
3. 顶栏
4. 首页 hero 区
5. 首页卡片 / 公告 / 文件夹区
6. 设置页卡片 / 输入框 / 开关

## 灯泡入口设计

### 行为变化

当前“灯泡”在导航配置里还是普通项。

本次调整后：

- 灯泡点击不再改变 `activePage`
- 点击只触发 `theme` 切换

### 交互反馈

为了避免用户不确定灯泡是否生效，灯泡需要体现当前主题状态。

第一版建议：

- 保留灯泡文案不变
- 点击后立即刷新整套配色
- 灯泡本身可根据当前主题显示 active 态

是否要在灯泡项旁显示 `白天 / 夜间` 文本，本次不强制。

## localStorage 设计

建议使用单一 key，例如：

- `xnnehanglab.theme`

行为：

1. 初始化读取该 key
2. 若值为 `night` 或 `day` 则直接采用
3. 否则回退到 `night`
4. 每次切换后立即写回

## 测试范围

至少需要覆盖：

1. 默认主题行为
2. 点击灯泡后主题值切换
3. 根节点 `data-theme` 跟着切换
4. `localStorage` 被正确写入
5. 刷新后能从 `localStorage` 恢复主题

## 明确不做

本次不做以下内容：

- 不接 Tauri 配置持久化
- 不做系统主题自动跟随
- 不做第三套主题
- 不做主题编辑器
- 不重写现有页面结构

- `tokens.css`

- 对应测试

## 预期结果

完成后应达到：

- 应用在 `night` / `day` 间切换时，整体风格同步变化
- 用户点击灯泡即可切换
- 关闭后重开仍保留上次主题
- 改动集中在壳层和主题变量，不影响现有页面结构与业务入口
